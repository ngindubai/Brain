# Week Plan - Mon 6 to Fri 10 July 2026

Five working days (UAE Mon-Fri). Roughly 3.5 days of estimated build work plus admin, so the week is full but achievable if the order holds. The one hard dependency: the humaniser fix comes before any new SEO content ships, because publishing through a failing process just adds pages you will have to redo.

## The five workstreams

| # | Workstream | Est. effort | Gates / depends on |
|---|---|---|---|
| 1 | Humaniser solution (AI-detection fix) | ~0.5 day research + setup | Blocks SEO content publishing |
| 2 | SEO tidy-up finish (compare + lead-gen) | ~0.5 day | Needs humaniser done before content resumes |
| 3 | Routine prompt + CLAUDE.md rewrite (learn-from-builds) | ~0.5 day admin | None |
| 4 | Atlas build finish + test, integrate Moonie's plan | ~1.5 days | None, but heaviest |
| 5 | Orwell Tours build (rename first) | ~1.5 days | Rename decision needed up front |

That totals ~4.5 days of work against 5 available. Atlas and Tours cannot both fully complete in one week alongside the rest. The plan front-loads the blockers and the quick wins, protects two clear days for Atlas, and starts Tours, with an honest note that Tours likely spills into the following week.

## Monday 6 July - Unblock and clear the small stuff

The theme today is removing blockers so the rest of the week runs clean. Humaniser first because it gates content, then the two half-day admin jobs.

### Morning: Humaniser solution (workstream 1)
- [ ] Define the actual failure. Which detector is the content failing (Originality.ai, GPTZero, Copyleaks, Turnitin)? Get one real failing sample and its score before researching fixes.
- [ ] Research the current state of humanising. Note: your earlier position was to reframe around genuine voice quality rather than gaming detectors. That was right in principle but it is now failing in practice, so this pass needs a working process, not a philosophy. Both can be true: build real voice AND pass detection.
- [ ] Decide the approach: prompt-level changes to the writing routine (burstiness, sentence-length variance, first-person specifics, real data points), a post-generation humanising pass, or a third-party API in the pipeline. Weigh cost and reliability.
- [ ] Produce a testable spec: take one failing article, apply the new process, re-run the same detector, record the before/after score. No spec ships without a passing re-test.

### Afternoon: Routine prompt + CLAUDE.md rewrite (workstream 3)
- [ ] Open Claude Code in each relevant repo (the daily-brief Brain repo, and the content-build repos for the sites).
- [ ] Paste the current routine prompts in and ask Claude Code to rewrite both the prompts and the CLAUDE.md files so the system learns from past builds and produces clean builds going forward. Frame it explicitly: "read the last N builds, find what went wrong or was redone, encode the fix as a standing rule."
- [ ] The architect-loop research just filed in the vault ([[Architect Loop]] if built, or the research note) is directly relevant here: the frozen-gates and repo-as-memory discipline is exactly what "learn from builds and build clean" means. Pull the useful rules from it into the CLAUDE.md rewrites.
- [ ] Commit the rewritten prompts and CLAUDE.md files. Note in each what changed and why.

**End-of-day check:** humaniser has a tested spec, routine prompts and CLAUDE.md files are rewritten and committed. Both blockers cleared.
