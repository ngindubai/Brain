## Story 1: Semrush MCP connector works, but account is out of API units — Sensor still unreadable
**Source:** Direct tool test, this session | mcp__Semrush__tracking_research, mcp__Semrush__execute_report
**Related vault notes:** [[SEO Portfolio]], [[Comparison Network]]

The Semrush MCP connector referenced in yesterday's open loop is now available and was tested properly today, for the first time. It authenticates correctly — the response confirms "the user has an active Semrush subscription" — which rules out a connector/auth problem. The blocker is different: the account does not have enough API units to complete a `tracking_research` call, and the tool response explicitly declines to suggest workarounds, pointing only to `semrush.com/mcp-access` to buy more units.

This matters for how the open loop gets closed out. The last two days' failures were "can't reach the data" (dashboard fetch blocked, web search too thin). Today's failure is "can reach it, but the account can't afford the call." That's a billing/plan decision, not a tooling one — worth flagging because it's a five-minute fix if Gareth wants live Sensor and position-tracking data available to Claude going forward, versus a dead end if the connector itself needs replacing.

One structural point worth logging for later: the toolset is organised around discovery tools (`keyword_research`, `organic_research`, `overview_research`, `backlink_research`, `tracking_research`, `siteaudit_research`, `subdomain_research`, `subfolder_research`, `projects_research`) each exposing `get_report_schema` → `execute_report`. `tracking_research` (position tracking / ranking changes / visibility scores) is the closest thing to a Sensor-equivalent in this toolset — Semrush Sensor itself is a separate consumer-facing product, not obviously exposed as its own MCP report. Because the API-unit error fired before a schema could even be pulled, it's not yet confirmed whether `tracking_research` can produce a Sensor-style aggregate volatility number or only per-domain/per-project tracking (which would need a Semrush project already configured for portfolio domains). That's worth resolving once units are available, since it changes whether this becomes the daily Sensor source or just a project-tracking add-on.

Public Sensor score for today could not be confirmed by any route: the dashboard page is JS-rendered and returns only navigation boilerplate to a fetch, and web search returned only explainer content about how Sensor works, not today's number. No substitute figure is being reported — better to say plainly it's unavailable than guess.
